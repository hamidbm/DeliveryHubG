# Codex Implementation Spec: Server-Side Caching for the Work Item Tree ## Goal Improve responsiveness of the left work-item tree by avoiding repeated full database reads every time the user changes filters. The desired behavior is: - fetch work items once - keep them in a **server-side cache** - apply most filter changes against the cached dataset - invalidate or refresh the cache when relevant work items change This cache should improve UX without breaking correctness. --- # Problem Statement Today, the work-item tree appears to re-query MongoDB repeatedly when: - changing filters - changing scope - expanding or collapsing tree regions - possibly switching tabs or routes that reuse the same work-item dataset That leads to: - visible delay in the left tree - unnecessary database load - repetitive reconstruction of the same work-item hierarchy - poor perceived performance in sample/demo scenarios with a lot of generated data The app should instead support a **read-through server cache** for work items. --- # Design Principles ## 1. Cache on the server, not only on the client The primary cache should be on the server side. Reasons: - avoids repeated DB reads across requests - keeps sensitive query logic centralized - allows one canonical tree/hierarchy build path - supports multiple client filter changes without reloading the base dataset Client-side caching can still exist, but it is not the primary design target here. --- ## 2. Cache the base dataset, not every filter permutation Do **not** cache every possible filtered combination. That would explode in complexity. Instead: - cache the **base work-item dataset** for a scope - cache the **derived hierarchy/tree** - apply lightweight filters in memory on top of the cached structure This is the correct tradeoff. --- ## 3. Scope-aware caching The cache must be keyed by the logical work-item scope. Examples: - bundle scope - application scope - possibly roadmap or milestone scope if the current screen supports them A cache entry must not mix work items from different scopes. --- ## 4. Versioned invalidation, not manual guessing only The safest design is: - cache by scope - attach a version marker or freshness metadata - invalidate when work items in that scope change Do not rely only on TTL. TTL can exist as a safety fallback, but invalidation should be explicit whenever possible. --- # What Should Be Cached ## Cache level 1: Raw work-item records Cache the full list of work items for a scope. This dataset should include all fields needed for: - building the tree - filtering - sorting - counts/badges - assignment/status display in the tree Do not cache incomplete records if later filters require fields that are missing. --- ## Cache level 2: Precomputed hierarchy Also cache the derived tree structure or indexed lookup maps. For example: - `byId` - `childrenByParentId` - `roots` - `itemsByType` - `itemsByMilestone` - `itemsBySprint` - `itemsByStatus` - `itemsByAssignedTo` - flattened ordered tree if needed This avoids rebuilding the hierarchy on every request. --- ## Cache level 3: Optional filtered result memoization This is optional and should be limited. You may memoize a small number of recent filtered results per scope, but only if it is cheap and bounded. Recommended first implementation: - cache only the base dataset and hierarchy - perform filtering in memory per request - do not add filter-permutation memoization yet --- # Required Architecture ## Introduce a dedicated cache service Create a new service: - `src/services/workItemCache.ts` This service becomes the only place that owns: - cache keys - cache storage - dataset hydration - invalidation - tree building reuse Do not scatter caching logic across route handlers. --- # Service Responsibilities The service should export functions like the following. ## 1. `getWorkItemTreeData(scope, options?)` This is the main read entry point. It should: 1. compute cache key from scope 2. check cache 3. if cache hit and valid, return cached data 4. if miss or stale, fetch from DB 5. build hierarchy/indexes 6. store in cache 7. return result ### Suggested return shape ```ts type WorkItemTreeCachePayload = { scopeKey: string; generatedAt: number; sourceVersion: string | number | null; items: WorkItemRecord[]; indexes: { byId: Record<string, WorkItemRecord>; childrenByParentId: Record<string, string[]>; rootIds: string[]; }; stats: { total: number; epics: number; features: number; stories: number; tasks: number; }; }; ``` --- ## 2. `filterWorkItemTreeData(payload, filters)` Applies in-memory filtering against cached payload. It should support existing filters already used by the UI. Examples may include: - type - status - assignee - team - milestone - sprint - text search - application - bundle - health - flagged-only - show-completed / hide-completed Important: Filtering must preserve valid tree semantics. That means the filter layer should define whether to: - include only direct matches, or - include ancestors of matches so the tree remains navigable Recommended behavior: - include matching nodes - include ancestor chain of matching nodes - optionally include descendants only when that is already the current UI behavior --- ## 3. `invalidateWorkItemScope(scope)` Invalidate one scope. This should be called whenever a write operation changes work items in that scope. Examples: - create epic/feature/story/task - update status - update assignment - reorder - delete - bulk edit - generate delivery plan - reset demo data - import/seed sample data --- ## 4. `invalidateAllWorkItemCaches()` Administrative fallback for large reset operations. Use when: - sample data reset runs - full bootstrap changes work items heavily - major batch job modifies many scopes --- ## 5. `primeWorkItemScope(scope)` Optional helper to proactively warm cache. Useful after: - delivery-plan generation - sample data generation - initial page load for likely active scope This is optional in the first implementation but recommended. --- # Cache Storage Strategy ## Recommended first implementation: in-process memory cache Use an in-memory cache first. For example: - module-level `Map<string, CacheEntry>` - or `lru-cache` if already available or acceptable to add This is the simplest and best first step. ### Why this is acceptable - easy to implement - immediate performance win - good enough for a single-node deployment - good enough for demo/sample environments - low risk ### Caveat In-memory cache is per server instance. So in multi-instance deployments: - each instance has its own cache - invalidation is local only unless a distributed cache is added later That is acceptable for phase 1 if documented. --- ## Recommended cache entry shape ```ts type WorkItemTreeCacheEntry = { key: string; payload: WorkItemTreeCachePayload; createdAt: number; lastAccessedAt: number; expiresAt: number; }; ``` --- ## Recommended TTL Use a modest TTL as a safety net, for example: - 60 seconds - or 120 seconds TTL is not the main correctness mechanism. It is only a fallback. Explicit invalidation should still happen on writes. --- ## Recommended bounds Prevent unbounded memory growth. Examples: - max 50 scope entries - evict least recently used entries - if using a plain `Map`, implement simple pruning on insert Recommended first version: - max 25 or 50 scope entries - remove oldest or least recently accessed when full --- # Cache Key Design ## Key by logical scope, not by filter state Use a normalized key such as: ```ts workitems:scopeType=BUNDLE:scopeId=abc123 ``` or: ```ts workitems:BUNDLE:abc123 ``` Do not include filter state in the primary cache key. --- ## Include optional view mode only if dataset shape differs materially If one route needs minimal fields and another needs full hierarchy fields, do not create separate payload shapes unless necessary. Prefer one canonical “tree payload” per scope. --- # Fetch Strategy ## Single-query hydration per scope When loading work items for a scope, fetch all relevant work items in one DB pass if possible. The server should avoid fetching epics, then features, then stories, then tasks in separate repeated passes if one efficient query can do it. The ideal dataset includes: - all work items in scope - required display fields - parentId - work item type - status - assignment fields - ordering fields - milestone/sprint/app references if needed by filters - any flags currently shown in the tree --- ## Add projection discipline Do not fetch massive unused fields if some fields are large. For example, if work items include large generated descriptions or AI payloads not needed by the tree, exclude them from the tree cache. The cache payload should contain the fields needed by the tree and filters, not every field in the document. Create a dedicated projection for tree hydration. --- # Tree-Building Rules ## Build once per cached payload When cache is refreshed: 1. fetch raw work items 2. normalize records 3. build parent-child relationships 4. build root list 5. build lookup maps 6. store result in cache Then future requests reuse this. --- ## Preserve hierarchy integrity If a child references a missing parent: - do not crash - treat as orphan - include in a safe orphan bucket or promote to root - optionally log warning --- ## Sorting Sort once during tree build using the same canonical ordering the UI expects. Examples may include: - type ordering - rank/sequence - createdAt fallback - name fallback Do not re-sort the entire dataset on every filter change unless necessary. --- # Filtering Rules ## Filters should run in memory against cached payload This is the core behavior change. When the user changes filters: - do not re-query MongoDB - do not rebuild the hierarchy from scratch - apply filter logic against the cached payload --- ## Preserve ancestors of matching nodes Example: - a task matches filter - its story, feature, and epic should remain visible so the tree makes sense Implement a helper: ```ts collectVisibleNodeIds(matches, byId, parentLookup) ``` This should include: - all matching ids - all ancestor ids up to root Optional descendants should be included only if current UI requires them. --- ## Text search If the filter includes free-text search: - perform it in memory over cached records for the first implementation - search fields may include: - title/name - key - assignedTo - tags/labels - milestone name if already hydrated - use case-insensitive contains matching Do not implement expensive fuzzy search in phase 1. --- # Invalidation Strategy ## Explicit invalidation is mandatory Every write path that mutates work items must invalidate the relevant scope cache. This includes: - create work item - update work item - delete work item - reorder work item - change parent - assign/unassign - change status - bulk update - create or delete milestones if tree filtering depends on milestone refs - create or delete sprints if tree filtering depends on sprint refs - delivery plan generation - sample data generation/reset --- ## Where to call invalidation Do not call invalidation from UI code. Call it from the server-side services or API handlers that perform writes. Examples: - `workItemService` - roadmap planning services - delivery-plan creation flow - sample data installer/reset path --- ## Scope invalidation helper Introduce a reusable helper that resolves scope from a work item or write payload. Example: ```ts await invalidateWorkItemScope({ scopeType: 'BUNDLE', scopeId: bundleId, }); ``` If a write affects multiple scopes, invalidate all relevant scopes. --- # Write-Through vs Invalidate-Only ## Recommended first implementation: invalidate-only When a write happens: - do not try to surgically mutate the cached hierarchy - simply invalidate the scope entry Then the next read will rebuild cleanly. This is simpler and much safer. ### Why not write-through first? Because the tree is hierarchical and filtered in many ways. Mutating cached structure in place is more error-prone. Invalidate-only is the correct first implementation. --- # Warmup Strategy ## Recommended warmup points After these flows, prime the relevant cache: ### 1. Delivery plan generation After `createDeliveryPlan(...)`, prime the affected bundle scope. ### 2. Sample data generation After demo scenario installation, prime all generated bundle scopes. ### 3. Initial screen load Optional. If the route already knows the current active scope, it may call the cached read path immediately. This reduces first-interaction lag. --- # API and Service Refactor Requirements ## Existing tree endpoint or loader must switch to cached service Wherever the left tree currently gets its server data, that code must be refactored to call: ```ts getWorkItemTreeData(scope) ``` instead of directly calling DB query logic every time. Then apply: ```ts filterWorkItemTreeData(payload, filters) ``` before returning the response. --- ## Keep DB query logic centralized Do not duplicate the raw query in multiple places. Create one internal function such as: ```ts loadWorkItemsForTreeFromDb(scope): Promise<WorkItemRecord[]> ``` and let the cache service call it. --- # Data Types ## Suggested scope type ```ts type WorkItemScope = { scopeType: 'BUNDLE' | 'APPLICATION'; scopeId: string; }; ``` If the app currently supports more scope kinds, extend this type accordingly. --- ## Suggested filter type ```ts type WorkItemTreeFilters = { types?: string[]; statuses?: string[]; assignees?: string[]; milestoneIds?: string[]; sprintIds?: string[]; applicationIds?: string[]; search?: string; hideCompleted?: boolean; flaggedOnly?: boolean; }; ``` Adjust to actual filter model already used by the UI. --- # Observability Requirements ## Add lightweight cache metrics/logging Log enough to verify benefit. For example: - cache hit - cache miss - cache invalidated - cache rebuilt - rebuild duration - item count - filter duration Recommended first version: ```ts console.info('[workItemCache] hit', { key }); console.info('[workItemCache] miss', { key }); console.info('[workItemCache] rebuild', { key, count, ms }); console.info('[workItemCache] invalidate', { key, reason }); ``` If the app has a logger utility, use that instead. --- ## Optional diagnostics endpoint Optional admin-only endpoint: - `GET /api/admin/cache/workitems` Return: - cache keys - item counts - createdAt - expiresAt - hit/miss counters if tracked Useful during tuning, but not required for phase 1. --- # Error Handling ## Cache failure must not break the tree If cache logic throws: - log the error - fall back to direct DB load - return correct data Caching is an optimization, not a correctness dependency. --- ## Corrupted cache entries If a cached payload is malformed: - discard it - rebuild from DB - continue safely --- # Recommended File Changes ## New file - `src/services/workItemCache.ts` ## Likely modified files Depending on actual code structure, Codex should update whichever files currently drive tree loading, likely among these kinds of files: - work item service - tree API route - work item query helpers - delivery plan creation service - sample data generation/reset services - work item mutation APIs The rule is simple: - all tree reads go through cache service - all work-item-affecting writes invalidate cache service --- # Suggested Internal Implementation ## Module-level cache store ```ts const CACHE_TTL_MS = 60_000; const MAX_SCOPE_ENTRIES = 50; const workItemTreeCache = new Map<string, WorkItemTreeCacheEntry>(); ``` --- ## Main read flow ```ts export async function getWorkItemTreeData(scope: WorkItemScope): Promise<WorkItemTreeCachePayload> { const key = toScopeKey(scope); const now = Date.now(); const cached = workItemTreeCache.get(key); if (cached && cached.expiresAt > now) { cached.lastAccessedAt = now; return cached.payload; } const items = await loadWorkItemsForTreeFromDb(scope); const payload = buildTreePayload(scope, items); workItemTreeCache.set(key, { key, payload, createdAt: now, lastAccessedAt: now, expiresAt: now + CACHE_TTL_MS, }); pruneCacheIfNeeded(); return payload; } ``` --- ## Invalidation flow ```ts export async function invalidateWorkItemScope(scope: WorkItemScope): Promise<void> { workItemTreeCache.delete(toScopeKey(scope)); } ``` --- ## Filter flow ```ts export function filterWorkItemTreeData( payload: WorkItemTreeCachePayload, filters: WorkItemTreeFilters ): FilteredTreeResult { const matchingIds = findMatchingIds(payload, filters); const visibleIds = includeAncestors(payload, matchingIds); return buildFilteredTree(payload, visibleIds); } ``` --- # Phase Plan ## Phase 1: Base server cache Implement now: - in-process server cache - per-scope payload caching - in-memory filtering - explicit invalidation on all work-item writes - optional warmup after delivery-plan generation/sample generation This should solve the current UX issue. --- ## Phase 2: Smarter optimization Later, if needed: - LRU cache implementation - distributed cache such as Redis - filter-result memoization - cache metrics dashboard - event-driven cross-instance invalidation Do not block phase 1 on these. --- # Acceptance Criteria Codex implementation is complete only if all of the following are true. ## Functional - changing work-item tree filters no longer triggers a full DB reload for each change - work items are fetched once per scope and reused from server cache - filtering runs against cached data - tree still reflects writes after invalidation ## Correctness - tree hierarchy remains correct - matching descendants still show required ancestors - cache invalidates after work-item create/update/delete flows - sample data generation and reset invalidate or warm relevant scopes correctly ## Performance - repeated filter changes are noticeably faster - DB reads for the same scope are reduced significantly - tree rebuild happens only on cache miss or invalidation ## Safety - cache failure falls back to DB load - memory usage remains bounded - cache entries expire by TTL even if invalidation is missed --- # Final Instruction to Codex Implement this as a **server-side, scope-based, invalidate-on-write cache** for the work-item tree. Do not try to cache every filtered permutation. Cache the **base scoped dataset and hierarchy**, then filter in memory.
