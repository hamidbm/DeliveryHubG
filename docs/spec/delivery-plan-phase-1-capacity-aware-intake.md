# DeliveryHub Spec: Phase 1 - Capacity-Aware Delivery Plan Intake and Derived Milestone Duration ## Document Purpose This document defines the exact implementation instructions for Codex for **Phase 1** of the Delivery Plan enhancement in the **Work Items** module. This phase covers: 1. Adding **capacity-aware inputs** to the **Generate Delivery Plan** intake flow 2. Making **milestone duration derived automatically** when the strategy is `AUTO_DISTRIBUTE` 3. Adding **project size presets** that prefill structural planning defaults 4. Making **task generation configurable** 5. Making the **preview** capacity-aware 6. Persisting milestone target capacity during **plan creation** This spec is intended to be: - the implementation brief for Codex - a permanent engineering spec stored under `docs/spec/` --- ## Background The current implementation already supports the new multi-step flow for **Generate Delivery Plan**: - intake form - preview - create draft artifacts The current flow already generates: - roadmap phases - milestones - sprints - epics - features - stories - optional tasks - dependency skeletons - owner suggestions - milestone-to-roadmap linkage However, the current implementation still has these Phase 1 gaps: ### Gap 1: Intake form is not capacity-aware The current generator only applies capacity indirectly through `bundle_capacity` lookup in the backend. This means the plan is still largely structural, not capacity-driven. ### Gap 2: Milestone duration is not properly derived in the UI The wizard currently asks for milestone duration, but when `milestoneDurationStrategy = AUTO_DISTRIBUTE`, that value should be **derived automatically** from: - overall start - delivery end / go-live - milestone count The current backend slices milestones automatically, but the wizard does not clearly expose the derived milestone duration to the user. ### Gap 3: Complexity defaults are too static The generator currently relies mostly on `backlogShape`, `storiesPerFeatureTarget`, and `featuresPerMilestoneTarget`, but there is no high-level complexity preset such as `projectSize`. ### Gap 4: Tasks per story are hardcoded Today, when tasks are generated, the generator uses a hardcoded count of `2` tasks per story. This must become configurable. --- ## Important Product Decisions for Phase 1 These decisions are intentional and should be implemented exactly as specified. ### 1. Do not add `teamSize` to the intake form in Phase 1 Do **not** add `teamSize` yet. Reason: - it adds confusion for users - it is not needed for the first useful capacity model - the user explicitly found `teamCount` and `teamSize` confusing - Phase 1 should keep the capacity model simple ### 2. Use `deliveryTeams`, not `teamCount` To reduce ambiguity in the UI and payload, use the label and field name: - `deliveryTeams` This means: - the number of teams working in parallel on the scope ### 3. Velocity is not derived in Phase 1 Do **not** try to compute velocity from historical data in this phase. Instead: - accept velocity as a user input - provide a default - use it for capacity calculations ### 4. Capacity mode must support two simple models Support exactly these modes: - `TEAM_VELOCITY` - `DIRECT_SPRINT_CAPACITY` ### 5. Project size is a preset, not a computed truth `projectSize` is not something the system computes automatically in Phase 1. It is a user-selected planning preset that prefills structural defaults. ### 6. Milestone duration is derived when strategy is `AUTO_DISTRIBUTE` When `milestoneDurationStrategy = AUTO_DISTRIBUTE`: - the user should not manually edit milestone duration - the UI should display the derived milestone duration - the backend should compute and return the derived duration in preview metadata When `milestoneDurationStrategy = FIXED_WEEKS`: - the user enters `milestoneDurationWeeks` - the backend uses that value directly --- ## Current Implementation Surface Codex must work from the current implementation, not redesign from scratch. ### Current primary files in use - `src/components/GenerateDeliveryPlanWizard.tsx` - `src/services/deliveryPlanGenerator.ts` - `src/app/api/work-items/plan/preview/route.ts` - `src/app/api/work-items/plan/create/route.ts` ### Existing legacy flow that should not be broken There is also an older intake flow: - `src/components/WorkPlanIntakeModal.tsx` - `src/app/api/work-items/plan/route.ts` That appears to be a legacy / alternate path. Phase 1 should **not** remove it unless Codex confirms it is unused. It may be left unchanged for now. ### Existing relevant behavior in the generator Current `deliveryPlanGenerator.ts` already does the following: - validates dates - computes milestones - computes roadmap phases - computes sprints - assigns sprints to milestones - generates artifacts - optionally suggests milestone owners - optionally suggests work item owners - optionally generates dependency skeleton - stores preview in `work_plan_previews` - creates actual milestones, roadmap phases, sprints, work items from preview Current capacity behavior is limited: - it calls `getBundleCapacity(scope.bundleId)` - if found, it derives milestone `targetCapacity` - if not found, it warns that no bundle capacity is configured This is the specific gap Phase 1 must fix. --- ## Phase 1 Objectives Codex must implement all of the following. ### Objective A: Add capacity planning inputs to the wizard Add a new capacity section to the intake form. ### Objective B: Add project size preset behavior Add a simple project size selection that prefills structural generation defaults. ### Objective C: Derive milestone duration automatically when auto-distribute is selected The intake form must show this derived value clearly. ### Objective D: Make preview capacity-aware Preview must return and display capacity numbers, not just generated structure. ### Objective E: Persist milestone target capacity during plan creation The target capacity computed during preview must survive creation and be stored on generated milestones. ### Objective F: Make tasks-per-story configurable Task count must no longer be hardcoded to `2`. --- ## Data Model Changes ## Extend `DeliveryPlanInput` Modify `src/services/deliveryPlanGenerator.ts` and add the following fields to `DeliveryPlanInput`: ```ts capacityMode?: 'TEAM_VELOCITY' | 'DIRECT_SPRINT_CAPACITY'; deliveryTeams?: number; sprintVelocityPerTeam?: number; directSprintCapacity?: number; projectSize?: 'SMALL' | 'MEDIUM' | 'LARGE' | 'ENTERPRISE'; tasksPerStoryTarget?: number; ``` ### Semantics - `capacityMode` - Determines how total sprint capacity is computed - `deliveryTeams` - Number of delivery teams working in parallel - `sprintVelocityPerTeam` - Story points one team can complete in one sprint - `directSprintCapacity` - Total story points across all teams in one sprint - Used only when `capacityMode = DIRECT_SPRINT_CAPACITY` - `projectSize` - Prefill preset for structural defaults - Must remain overridable - `tasksPerStoryTarget` - Number of tasks to generate under each story when tasks are enabled --- ## Extend `DeliveryPlanPreview` Modify the preview response shape to include new metadata. Add these sections: ```ts capacitySummary?: { mode: 'TEAM_VELOCITY' | 'DIRECT_SPRINT_CAPACITY' | 'BUNDLE_CAPACITY_FALLBACK' | 'NONE'; sprintCapacity: number | null; milestoneCapacities: Array<{ milestoneIndex: number; sprintCount: number; targetCapacity: number | null; }>; }; derived?: { milestoneDurationDays?: number | null; milestoneDurationWeeks?: number | null; }; ``` ### Notes - `capacitySummary.mode` should reflect the actual source used - if the preview used bundle fallback, return `BUNDLE_CAPACITY_FALLBACK` - if nothing is available, return `NONE` --- ## Files Codex Must Modify Primary files: - `src/components/GenerateDeliveryPlanWizard.tsx` - `src/services/deliveryPlanGenerator.ts` - `src/app/api/work-items/plan/preview/route.ts` - `src/app/api/work-items/plan/create/route.ts` Likely also: - `src/types.ts` if shared milestone or preview types need extension - `src/components/WorkItems.tsx` only if required for preview typing or dialog integration Do **not** begin the planning-engine refactor in this phase. Fix milestone duration UX in the wizard The milestone duration UX must change. ### Current problem The wizard currently always shows a milestone duration input, but when strategy is `AUTO_DISTRIBUTE`, the duration should be derived. ### Required behavior When `milestoneDurationStrategy = AUTO_DISTRIBUTE`: - the editable input for `milestoneDurationWeeks` must be hidden or disabled - instead, show a read-only derived value such as: - `Derived milestone duration: 3.5 weeks` - if there is insufficient date information to compute it, show a neutral helper message When `milestoneDurationStrategy = FIXED_WEEKS`: - the numeric input for `Milestone Duration (weeks)` is editable - this value is passed to the backend ### Formula to use in the wizard Use: - `effectiveStart = plannedStartDate || devStartDate` - `effectiveEnd = goLiveDate` - `milestoneCount` Then compute: ```ts overallWindowDays = max(0, differenceInDays(effectiveEnd, effectiveStart)) derivedMilestoneDurationDays = overallWindowDays / milestoneCount derivedMilestoneDurationWeeks = derivedMilestoneDurationDays / 7 ``` Round for display only, for example to one decimal place. ### Validation behavior If one of the required dates is missing: - do not crash - show no derived duration - allow normal validation to continue --- ## 7.
