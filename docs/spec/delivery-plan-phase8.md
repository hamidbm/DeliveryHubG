# DeliveryHub Spec: Phase 8 – Probabilistic Forecasting ## Document Purpose This document defines the implementation instructions for **Phase 8: Probabilistic Forecasting** in DeliveryHub. Phase 8 extends the deterministic forecasting introduced in Phase 7 by adding **probability-based delivery forecasting** for milestones, plans, and portfolios. This phase introduces: - milestone forecast distributions - percentile-based forecast dates - probability of on-time delivery - portfolio-level uncertainty rollups - probabilistic forecast visualization This document serves as: - the **implementation guide for Codex** - the **permanent specification stored in `docs/spec/`** --- # Context After Phases 1–7, DeliveryHub supports: - capacity-aware planning - roadmap intelligence - simulation - portfolio analytics - deterministic predictive forecasting The next step is to move from single-value forecast outputs to **distribution-based forecast outputs**. Phase 7 can answer: - best case / expected / worst case Phase 8 must answer: - what is the P50 date? - what is the P75 date? - what is the P90 date? - what is the probability of hitting the planned end date? - which plans have the greatest uncertainty? --- # Objectives Phase 8 introduces the following capabilities: 1. Milestone forecast distributions 2. Percentile delivery dates 3. Probability of on-time delivery 4. Plan-level uncertainty summaries 5. Portfolio-level uncertainty summaries 6. Probabilistic overlays in roadmap and portfolio views The goal is to give users a more realistic understanding of delivery uncertainty. --- # Non-Goals Phase 8 does **not** include: - automated schedule optimization - AI/ML forecasting models - dynamic resource balancing - cross-portfolio financial forecasting Phase 8 should remain: - explainable - testable - deterministic enough to validate in code - grounded in the intelligence model already present in DeliveryHub --- # High-Level Modeling Approach Use a **Monte Carlo-style sampling approach** with bounded, explainable assumptions. Each milestone already has intelligence signals, including: - utilization - risk level - blocked item count - dependency pressure - readiness - confidence - duration These will determine the amount of uncertainty injected into milestone finish-date simulations. --- # Forecast Distribution Model Each milestone should produce a probabilistic forecast distribution summarized as percentile dates. ## New Types in `src/types.ts` ### Milestone Probabilistic Forecast ```ts MilestoneProbabilisticForecast = { milestoneId: string plannedEndDate: string p50Date: string p75Date: string p90Date: string onTimeProbability: number uncertaintyLevel: 'LOW' | 'MEDIUM' | 'HIGH' } ``` ### Plan Probabilistic Forecast Summary ```ts PlanProbabilisticForecastSummary = { planId: string milestonesAnalyzed: number lowConfidenceMilestones: number averageOnTimeProbability: number averageP50SlipDays: number averageP90SlipDays: number } ``` ### Portfolio Probabilistic Forecast Summary ```ts PortfolioProbabilisticForecastSummary = { plansAnalyzed: number milestonesAnalyzed: number averageOnTimeProbability: number highUncertaintyMilestones: number averagePortfolioP90SlipDays: number } ``` --- # Probabilistic Forecasting Engine Create: ```text src/services/probabilisticForecastingEngine.ts ``` ## Responsibilities - generate simulated milestone completion dates - compute percentile outputs - compute on-time delivery probability - aggregate plan-level summaries - aggregate portfolio-level summaries --- # Core Functions ## 1. `sampleMilestoneFinishDates` ```ts function sampleMilestoneFinishDates( milestone, intelligence, options? ): Date[] ``` This function should: - generate multiple simulated finish dates - use bounded randomized delay/spread based on milestone intelligence - keep simulation reproducible and testable ### Important requirement Use a deterministic pseudo-random approach with seed support. Do **not** use uncontrolled randomness that makes tests unstable. Accepted approach: - seeded RNG - default deterministic seed if one is not provided - configurable sample count --- ## 2. `computeMilestoneProbabilisticForecast` ```ts function computeMilestoneProbabilisticForecast( milestone, intelligence, options? ): MilestoneProbabilisticForecast ``` This function should: - run milestone sampling - sort simulated finish dates - compute: - P50 - P75 - P90 - on-time probability - uncertainty level --- ## 3. `computePlanProbabilisticForecast` ```ts function computePlanProbabilisticForecast(plan): { milestoneForecasts: MilestoneProbabilisticForecast[] summary: PlanProbabilisticForecastSummary } ``` This function aggregates milestone probabilistic forecasts for a single plan. --- ## 4. `computePortfolioProbabilisticForecast` ```ts function computePortfolioProbabilisticForecast(plans): { planForecasts: Array<{ planId: string summary: PlanProbabilisticForecastSummary }> portfolioSummary: PortfolioProbabilisticForecastSummary } ``` This function aggregates probabilistic forecasts across requested plans. --- # Sampling Heuristics The simulation must be explainable and bounded. ## Base spread Start with a base uncertainty derived from milestone duration: ```ts baseSpreadDays = max(2, round(milestoneDurationDays * 0.15)) ``` ## Risk adjustments Increase spread according to milestone intelligence. ### Utilization adjustment ```ts if utilizationPercent > 1.0 => +3 spread days if utilizationPercent > 1.2 => +6 spread days ``` ### Dependency pressure adjustment ```ts if dependencyInbound > 3 => +2 spread days if dependencyInbound > 6 => +5 spread days ``` ### Blocked items adjustment ```ts if blockedItemCount > 5 => +3 spread days if blockedItemCount > 10 => +6 spread days ``` ### Readiness adjustment ```ts NOT_READY => +5 spread days PARTIAL => +2 spread days READY => +0 ``` ### Confidence adjustment ```ts LOW confidence => +4 spread days MEDIUM confidence => +2 spread days HIGH confidence => +0 ``` --- # Sampling Distribution For Phase 8, use a simple bounded asymmetric delay model. ### Recommended approach Generate simulated finish dates by: - centering around planned end date - allowing a smaller early-completion range - allowing a larger late-completion range - biasing delay spread upward for risky milestones Example conceptual model: - early finish window: up to `baseSpreadDays * 0.5` - late finish window: up to `baseSpreadDays + riskAdjustments` This reflects the reality that milestones are usually more likely to slip late than finish significantly early. ### Important requirement The distribution must be: - reproducible - explainable - bounded Do not introduce advanced statistics or external ML libraries in this phase. --- # Percentile Calculation After sampling N finish dates: - sort ascending - compute: - P50 - P75 - P90 ### Sample count Use a reasonable default sample count such as: ```ts sampleCount = 500 ``` Allow this to be overridden for tests if needed. ### On-time probability ```ts onTimeProbability = simulatedDatesOnOrBeforePlannedEnd / totalSampleCount ``` Represent as a number in `[0, 1]`. --- # Uncertainty Level Classification Compute from spread and on-time probability. Suggested rules: ```ts HIGH uncertainty: - P90 - P50 > threshold or - onTimeProbability < 0.5 MEDIUM uncertainty: - moderate spread or - onTimeProbability between 0.5 and 0.8 LOW uncertainty: - tight spread and - onTimeProbability >= 0.8 ``` Codex may tune thresholds slightly, but must keep them simple and documented. --- # API Endpoints Add probabilistic forecast APIs. ## Plan Probabilistic Forecast ```text GET /api/probabilistic-forecast/plan/[planId] ``` ### Supported plan ID format Use the same Phase 6 / Phase 7 plan identifier format: - `created:<id>` - `preview:<id>` ### Response ```ts { milestoneForecasts: MilestoneProbabilisticForecast[] summary: PlanProbabilisticForecastSummary } ``` --- ## Portfolio Probabilistic Forecast ```text POST /api/probabilistic-forecast/portfolio ``` ### Request ```ts { planIds: string[] } ``` ### Response ```ts { planForecasts: Array<{ planId: string summary: PlanProbabilisticForecastSummary }> portfolioSummary: PortfolioProbabilisticForecastSummary } ``` ### Scope rule Forecast **only** the explicitly requested plan IDs. Do not auto-include related plans. --- # Backend Data Sources Probabilistic forecasting must use **backend/domain intelligence**, not UI-layer view model logic. ### Required sources Use the same general approach as Phase 7: - milestone rollups - readiness evaluation - dependency pressure - persisted or preview milestone dates - milestone intelligence from backend/domain sources ### Important rule Do **not** import UI components or `roadmapViewModels` into the probabilistic forecasting engine. UI view models may consume probabilistic forecast outputs for rendering, but not the reverse. --- # UI Integration ## Roadmap UI Enhance roadmap views to surface probabilistic forecasting. Update: ```text src/components/roadmap/ExecutionBoardView.tsx src/components/roadmap/RoadmapTimelineView.tsx src/components/roadmap/RoadmapSwimlaneView.tsx ``` ### Execution Board For each milestone, show: - P50 date - P90 date - on-time probability - uncertainty badge Example: ```text Forecast: P50: Jun 12 P90: Jun 21 On-Time: 62% Uncertainty: Medium ``` ### Timeline View Add a probabilistic overlay beyond the planned end date. Recommended visual behavior: - planned bar remains the baseline - forecast range extends beyond planned end - show percentile markers or subtle band: - P50 - P75 - P90 The UI does not need to render a statistically perfect curve. A banded range is sufficient. ### Swimlane View Show compact probabilistic summary inside milestone cells: - P90 slip indicator - on-time probability - uncertainty badge --- ## Portfolio Dashboard Enhance: ```text src/components/portfolio/PortfolioDashboard.tsx ``` Add probabilistic summary cards such as: - average on-time probability - high-uncertainty milestones - average P90 slip days - plans with low confidence outlook If needed, add or update child components under `src/components/portfolio/`. --- # View Models Create or extend forecast view-model helpers. Acceptable locations: - extend existing roadmap/portfolio view-model files - or create a small shared helper if that is cleaner ### Suggested functions ```ts buildProbabilisticForecastViewModel(...) buildPortfolioProbabilisticForecastViewModel(...) ``` These helpers should: - transform API responses for rendering - compute display-friendly labels - avoid embedding core probabilistic logic in UI code --- # Testing Requirements Add deterministic tests for probabilistic forecasting. ## New test files ```text scripts/test-probabilistic-forecasting-engine.ts scripts/test-probabilistic-forecast-api.ts ``` ## Required test coverage ### Engine tests - milestone sampling is deterministic with a fixed seed - percentile outputs are computed correctly - on-time probability is computed correctly - higher-risk milestones produce wider forecast spreads than lower-risk milestones - uncertainty classification works ### API tests - `GET /api/probabilistic-forecast/plan/[planId]` - `POST /api/probabilistic-forecast/portfolio` - explicit plan ID handling works correctly - only requested plans are forecast in portfolio calls ### Regression requirement Ensure: ```text npm run test:api ``` continues to pass. --- # Documentation Requirements Create a new spec: ```text docs/spec/delivery-plan-phase-8-probabilistic-forecasting.md ``` Update wiki docs: ```text docs/wiki/Roadmap.md docs/wiki/Modules-WorkItems.md ``` The documentation must explain: - what probabilistic forecasting is - how percentile dates are interpreted - what on-time probability means - that results are heuristic and explainable, not ML-based - the supported APIs ### Required implementation status section At the end of implementation, Codex should add: ```md ## Implementation Status Status: Completed Validation: - probabilistic forecasting engine tests added - probabilistic forecast API tests added - `npm run test:api` passes Notes: - seeded deterministic sampling is used for test stability - supported plan ID formats are `created:<id>` and `preview:<id>` ``` --- # Acceptance Criteria Phase 8 is complete only when all of the following are true. ## Backend - probabilistic API tests exist - `npm run test:api` passes ## Documentation - Phase 8 spec exists and is accurate - wiki docs are updated --- # Out of Scope Phase 8 does not include: - full Monte Carlo research-grade modeling -
