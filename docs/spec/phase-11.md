# DeliveryHub Spec — Phase 11: Applications Module Deep Build (Application Portfolio Management) ## Overview Phase 11 expands the **Applications module** into a full **Application Portfolio Management (APM)** capability. The Applications module becomes the **system of record for application context**, feeding planning, optimization, forecasting, and portfolio analytics. Currently the module supports: - application entities - planning metadata - environment schedule - integration with delivery plan intake Phase 11 introduces richer application intelligence including: - application lifecycle management - environment lifecycle modeling - cross-application dependencies - ownership and vendor modeling - application criticality and tiering - release train modeling - topology relationships - delivery readiness indicators These capabilities allow DeliveryHub to understand **delivery context across the enterprise**, not just within a single plan. --- # Goals 1. Transform Applications into a **true APM module** 2. Provide context used by: - Work Items - Roadmaps - Forecasting - Optimization 3. Model **cross-application delivery risk** 4. Enable **portfolio-level delivery insights** 5. Maintain lightweight, developer-friendly data models --- # Architectural Principles Applications module becomes the **context layer** of the platform. Applications should: - define the delivery ecosystem - provide planning metadata - describe relationships between systems - feed optimization and forecasting models Applications **must remain lightweight** and avoid becoming a full CMDB. Focus on **delivery-relevant attributes only**. --- # Core Concepts ## Application A logical system or service that participates in delivery plans. Examples: - Payments API - Claims Processing - Customer Portal - Identity Service Applications are used to: - organize work items - model dependencies - understand release impact --- ## Application Portfolio A group of applications managed together. Examples: - Digital Banking Portfolio - Claims Platform Portfolio Portfolio groupings support: - portfolio roadmap - portfolio risk analysis - release coordination --- ## Environment Lifecycle Applications have environments such as: ``` DEV SIT INT QA PERF UAT STAGING PREPROD PROD ``` Environment lifecycles define: - deployment progression - integration windows - testing gates - go-live readiness --- ## Application Dependency Applications may depend on other applications. Example: ``` Customer Portal → Identity Service Claims API → Payments API ``` Dependencies impact: - milestone planning - delivery risk - forecast reliability --- # Data Model Changes ## Applications Collection (existing) Keep core identity fields simple. Example: ``` { _id, name, description, portfolioId, businessOwner, technicalOwner, vendorOwner, tier, lifecycleStatus, createdAt, updatedAt } ``` --- ## Application Portfolio Collection New collection: ``` application_portfolios ``` Example: ``` { _id, name, description, executiveOwner, createdAt, updatedAt } ``` --- ## Application Dependency Collection New collection: ``` application_dependencies ``` Example: ``` { _id, sourceApplicationId, targetApplicationId, dependencyType, criticality, createdAt } ``` Dependency types: ``` API DATA EVENT SHARED_INFRA ``` --- ## Application Lifecycle Collection New collection: ``` application_lifecycle ``` Example: ``` { applicationId, lifecycleStage, lifecycleOwner, lifecycleNotes, updatedAt } ``` Lifecycle stages: ``` ACTIVE MAINTENANCE SUNSETTING RETIRED ``` --- ## Environment Strategy Collection Extend existing planning metadata or create: ``` application_environment_strategy ``` Example: ``` { applicationId, environments: [ { name: "DEV", order: 1, description: "Developer integration environment" } ] } ``` --- # Application Metadata Enhancements Add fields to support delivery intelligence. New fields: ``` businessCriticality availabilityTier dataSensitivity regulatoryImpact operationalOwner releaseTrain ``` Example: ``` businessCriticality: HIGH availabilityTier: TIER1 releaseTrain: "DigitalBanking-Quarterly" ``` --- # Release Train Modeling New concept: ``` release_trains ``` Example: ``` { _id, name, cadence, portfolioId, description } ``` Example cadence: ``` Quarterly Monthly Biweekly ``` Applications can belong to release trains. --- # Cross-Application Impact Modeling Add ability to answer questions like: - Which applications are affected by this milestone? - Which applications share a release train? - Which applications depend on each other? Used by: - roadmap - forecasting - optimization --- # API Endpoints ## Applications Existing: ``` GET /api/applications POST /api/applications GET /api/applications/{id} PUT /api/applications/{id} ``` --- ## Application Portfolios ``` GET /api/application-portfolios POST /api/application-portfolios GET /api/application-portfolios/{id} PUT /api/application-portfolios/{id} ``` --- ## Application Dependencies ``` GET /api/application-dependencies POST /api/application-dependencies DELETE /api/application-dependencies/{id} ``` --- ## Environment Strategy ``` GET /api/applications/{id}/environment-strategy PUT /api/applications/{id}/environment-strategy ``` --- # UI Enhancements ## Applications Dashboard Add dashboard summarizing: - application count - lifecycle status - critical systems - dependency map --- ## Application Profile Page Enhance application detail page with tabs: ``` Overview Environments Dependencies Lifecycle Delivery Impact ``` --- ## Dependency Graph View Add graph visualization: ``` Application A → Application B Application B → Application C ``` Used to visualize system topology. --- # Integration with Work Items Applications become the **context anchor** for delivery plans. Work Items module will: - reference applications - load planning metadata - understand environment strategy - detect cross-application dependencies --- # Integration with Optimization Engine Optimization engine can leverage: - cross-application dependencies - release train constraints - environment lifecycle rules This improves scheduling decisions. --- # Integration with Forecasting Forecasting engine can include: - cross-system delivery risk - release train conflicts - dependency pressure between applications --- # Testing Add tests: ``` scripts/test-application-dependencies.ts scripts/test-application-portfolios.ts scripts/test-environment-strategy.ts ``` Validate: - dependency creation - lifecycle transitions - environment modeling - portfolio grouping --- # Acceptance Criteria Phase 11 is complete when: 1.
