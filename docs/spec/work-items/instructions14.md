# Next: RBAC Hardening v1 (Critical Actions Policy + Central Enforcement) ## Goal Make authorization consistent, explicit, and testable for all critical actions introduced so far: - milestone COMMIT/START/DONE transitions and overrides - milestone scope changes (assign/remove work items) when COMMITTED - over-capacity overrides (allowOverCapacity) - dependency creation/removal (BLOCKS in particular) - risk severity/criticality edits (if present) - admin-only configuration endpoints (confirm) This should remove scattered ad-hoc role checks and replace them with centralized policy functions. --- ## Part A — Define Roles + Policies in One Place In `src/services/authz.ts` (already exists), define explicit policy functions: - `canCommitMilestone(user)` - `canStartMilestone(user)` - `canCompleteMilestone(user)` - `canOverrideMilestoneReadiness(user)` - `canOverrideCapacity(user)` - `canEditCommittedMilestoneScope(user)` (adding/removing milestoneIds on work items) - `canCreateBlocksDependency(user, sourceItem, targetItem)` - `canRemoveBlocksDependency(user, sourceItem, targetItem)` Base rule set (v1): - Admin/CMO can do all critical actions - Bundle owner can: - edit scope and dependencies for work items within their bundle - cannot override readiness/capacity unless explicitly allowed (default: no) - Regular users: - can edit their assigned items but cannot do governance actions Use existing bundle assignments collection to resolve “bundle owner”. --- ## Part B — Enforce Policies in API Routes (Server-Side) Update APIs to use policy functions consistently: ### Milestones `PATCH /api/milestones/[id]` - enforce: - commit/start/done transitions - override requires canOverrideMilestoneReadiness ### Work items `PATCH /api/work-items/[id]` - enforce: - if milestoneIds changes and any target milestone is COMMITTED: - require canEditCommittedMilestoneScope - if allowOverCapacity is true: - require canOverrideCapacity - if links changes via /links endpoint or patch: - BLOCKS creation/removal requires canCreateBlocksDependency/canRemoveBlocksDependency `PATCH /api/work-items/bulk` - same enforcement rules: - allowOverCapacity gated - committed milestone scope gated ### Work item links endpoint (if present) `POST/DELETE /api/work-items/[id]/links` - enforce BLOCKS creation/removal via policy Return 403 with clear error codes like: - `FORBIDDEN_COMMIT_MILESTONE` - `FORBIDDEN_CAPACITY_OVERRIDE` - `FORBIDDEN_BLOCKS_DEPENDENCY` --- ## Part C — UI Consistency (Optional but recommended) Where UI shows controls: - Commit/Start/Done buttons - override modals Ensure they are enabled/disabled based on the same permissions returned from `/api/auth/me`. If `/api/auth/me` doesn't return role/permissions, extend it to include: - role - isAdminOrCMO - bundleOwnership list Avoid adding complicated permission matrices to the UI; UI can rely on simple flags. --- ## Part D — Add Audit Events for Forbidden Attempts (Optional) For critical endpoints, when returning 403, emit an event: - type: `security.forbidden` - include actor, action, resource, and why Keep this behind a flag if needed. --- ## Part E — Tests Add `scripts/test-rbac-critical-actions.ts`: - create users with roles: regular, bundle owner, admin - attempt: - commit milestone - capacity override assignment - create cross-bundle BLOCKS link - edit scope for COMMITTED milestone - assert expected 403/allowed results Add npm script: `test:rbac` --- ## Deliverables - central authz policy functions - updated API routes enforcing policies uniformly - clear 403 error codes - tests - optional forbidden attempt audit event
