# Next: COMMITTED Scope Change Workflow v1 (Approval + Audit + Notifications) ## Goal Introduce an approval workflow for scope changes to COMMITTED milestones: - adding/removing work items from a COMMITTED milestone becomes a request - requests can be approved/rejected by Admin/CMO (or bundle owner + Admin) - approved requests apply the change, with capacity re-validation - everything is auditable and notifies stakeholders Keep it MVP: one request type, simple UI. --- ## Part A — Data Model Add collection: `scope_change_requests` Schema: ```ts type ScopeChangeRequest = { _id: string; milestoneId: string; action: 'ADD_ITEMS' | 'REMOVE_ITEMS'; workItemIds: string[]; requestedBy: string; requestedAt: string; status: 'PENDING' | 'APPROVED' | 'REJECTED' | 'CANCELLED'; decidedBy?: string; decidedAt?: string; decisionReason?: string; // snapshot before: { committedPoints: number; targetCapacity?: number }; after?: { committedPoints: number; targetCapacity?: number }; allowOverCapacity?: boolean; } ``` Indexes: - { milestoneId: 1, status: 1, requestedAt: -1 } - { status: 1, requestedAt: -1 } --- ## Part B — API Endpoints Add: 1) Create request - POST /api/milestones/[id]/scope-requests body: { action: 'ADD_ITEMS'|'REMOVE_ITEMS', workItemIds: string[], allowOverCapacity?: boolean } Rules: - milestone must be COMMITTED - if allowOverCapacity is true, require canOverrideCapacity - compute before snapshot (rollup committedPoints + targetCapacity) Response: created request 2) List requests for milestone - GET /api/milestones/[id]/scope-requests?status=PENDING|... RBAC: - requester can see their own - Admin/CMO and bundle owners can see all for the milestone 3) Decide request - POST /api/milestones/[id]/scope-requests/[requestId]/decide body: { decision: 'APPROVE'|'REJECT', reason?: string } Rules: - Admin/CMO only to decide (v1) - When approving: - apply the milestoneIds change to the target work items - enforce capacity + missing estimate rules (same as normal assignment) - if over-capacity would occur and allowOverCapacity not set, reject approval with 409 and details - if approved, compute after snapshot - When rejecting: store reason Emit events: - milestones.scope.requested - milestones.scope.approved / milestones.scope.rejected Create notifications: - on request: notify Admin/CMO + bundle owners + milestone watchers - on decision: notify requester + watchers --- ## Part C — Integrate into Work Item Assignment Path Update /api/work-items/[id] and bulk: If milestone is COMMITTED and caller is NOT Admin/CMO: - block direct edits to milestoneIds with 409: - error: COMMITTED_SCOPE_REQUIRES_APPROVAL - include milestoneId and guidance link to create request If caller IS Admin/CMO: - allow direct edit as today (but emit event milestones.scope.directChanged) Optionally: require approval even for Admin (config flag). For v1, allow direct admin edits. --- ## Part D — UI (Milestone Planning) In Milestone Planning view: 1) Add a “Scope Requests” tab/panel for selected milestone: - shows pending requests list with: - action, count, requester, time, allowOverCapacity, status - Admin/CMO can approve/reject with reason - requester can cancel their pending request 2) When a non-admin user attempts to drag items into a COMMITTED milestone: - instead of failing hard, prompt: - “Create scope change request?” - allow selecting items in the action - Submit request to the API Keep UI minimal; no complex forms. --- ## Part E — Tests Add script: scripts/test-scope-requests.ts Cover: - non-admin cannot directly change committed scope (409) - create request succeeds - admin approve applies changes and emits events + notifications - approval blocked if capacity would be exceeded (when allowOverCapacity false) - reject records reason and notifies requester Add npm script: test:scope-requests Include in test:api. --- ## Deliverables - scope_change_requests collection + API endpoints - gating in work-item milestone assignment for non-admins in COMMITTED milestones - milestone planning UI panel for requests + approve/reject - events + notifications wired - tests + docs update (brief) `
