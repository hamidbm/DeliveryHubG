# Next: Visibility Model v1 (Access control for bundles/milestones/work items) ## Goal Implement consistent READ access rules across the app: - bundle profiles - work items (list/roadmap/tree/program) - milestone planning - feeds - notifications targets (avoid notifying users about things they cannot view) Keep it v1 and simple: - bundle-level visibility is the primary gate - milestone/work item visibility derives from bundle visibility - handle cross-bundle dependencies safely --- ## Part A — Define Visibility Rules (v1) Introduce bundle visibility levels: - PRIVATE: only Admin/CMO + bundle owners - INTERNAL: owners + watchers + any authenticated user (optional; choose one) - PUBLIC: anyone authenticated For v1, implement: - PRIVATE vs NON_PRIVATE only (two levels) if you want simpler. Default visibility: - INTERNAL (or NON_PRIVATE) for existing bundles. Who can view a bundle: - Admin/CMO always - bundle owners always - bundle watchers always - if visibility is NON_PRIVATE: any authenticated user can view Work item visibility: - visible if its bundle is visible - if work item has no bundleId, treat as INTERNAL unless configured Milestone visibility: - visible if user can view at least one bundle in its scope - if milestone spans multiple bundles, show only items from visible bundles Cross-bundle dependencies: - when showing a dependency to an item in a bundle the user cannot view: - do not reveal title/details - show as "Restricted item" with key only if key is not considered sensitive; else opaque placeholder - counts can remain but details hidden --- ## Part B — Data Model Extend Bundle with: ```ts visibility?: 'PRIVATE'|'INTERNAL'|'PUBLIC' ``` Add index: - bundles: { visibility: 1 } --- ## Part C — Central Access Helper Create: - src/services/visibility.ts Functions: - canViewBundle(user, bundleId) - canViewWorkItem(user, workItem) - filterVisibleWorkItems(user, items) - redactWorkItem(user, item) // returns minimal shape if restricted - canViewMilestone(user, milestoneId) (best effort) - filterVisibleEventsForFeed(user, feedItems) (or filter before transform) Use bundle ownership and watchers to decide. Cache bundle ownership/watchers lookup per request. --- ## Part D — Enforce in API Routes Update key APIs to enforce visibility: Work items: - GET /api/work-items (and list endpoints like board/tree/roadmap-intel/program-intel) - filter results to visible bundles - ensure aggregate counts roll up only visible items - GET /api/work-items/[id] - 404 if not visible (or 403) - Link-related endpoints: - when returning link summaries, redact restricted targets Milestones: - /api/milestones/rollups - /api/work-items/roadmap-intel - /api/program/intel - ensure scoping respects visibility Feed: - /api/feed - exclude events for restricted bundles/milestones - or redact titles/links Notifications: - When creating notifications for watchers/owners: - ensure recipient can view the referenced bundle/milestone/work item - otherwise skip recipient --- ## Part E — UI Behavior - If user cannot view a bundle: - hide it from bundle lists, program tables, filters - In cross-bundle blocker lists: - show restricted placeholders - In dependency graph: - restricted nodes should appear as "Restricted" with minimal metadata --- ## Part F — Migration + Defaults - Add a small migration script to set visibility=INTERNAL for existing bundles if field missing. --- ## Part G — Tests Add scripts/test-visibility.ts: - create bundles: PRIVATE and INTERNAL - create users: admin, owner, watcher, regular - assert: - regular cannot fetch private bundle work items (404/filtered) - watcher can view - program/intel and roadmap-intel exclude restricted bundles - dependency redaction works - notifications not sent to users lacking visibility Add npm script: test:visibility and include in test:api. --- ## Deliverables - bundle visibility field - visibility service helpers - API filtering/redaction across key endpoints - UI hides restricted bundles and redacts restricted dependencies - migration for defaults - tests
