# Next: Portfolio Capacity Planning v1 (Bundle Capacity Calendar + Overcommit Detection) ## Goal Provide a portfolio-level view that answers: - For each bundle, do we have enough capacity to deliver committed milestone scope? - Where are we overcommitted in upcoming weeks/sprints? - Which milestones are driving overcommit? - What are the recommended actions (reduce scope, move milestone, add capacity)? Keep it v1: - capacity is expressed in story points per sprint (or per week) - bundle capacity is configured manually in admin (no HR system integration) - computations are explainable, not black-box --- ## Part A — Bundle Capacity Model Add collection: - bundle_capacity Schema: ```ts type BundleCapacity = { _id: string; // bundleId bundleId: string; unit: 'POINTS_PER_SPRINT' | 'POINTS_PER_WEEK'; value: number; // e.g. 60 points per sprint updatedAt: string; updatedBy: string; }; ``` Index: unique { bundleId: 1 } Admin UI: - Admin → Operations (or Settings) → Bundle Capacity - table of bundles with capacity unit/value - edit inline Emit event: - bundle.capacity.updated --- ## Part B — Capacity Planning Computation Add service: - src/services/capacityPlanning.ts Inputs: - bundles in scope - bundle capacity config - committed milestones for each bundle - remainingPoints per milestone (from rollups/forecasting) - sprint schedule (active + upcoming sprints) OR a week-based calendar Output: ```ts type BundleCapacityPlan = { bundleId: string; capacity: { unit; value }; horizon: { startDate; endDate; buckets: 'SPRINT'|'WEEK'; count: number }; buckets: Array<{ key: string; // sprintId or week start startDate: string; endDate: string; capacityPoints: number; demandPoints: number; // sum of remaining points allocated to this bucket overBy: number; // max(0, demand-capacity) drivers: Array<{ milestoneId; name; demandPoints: number }>; }>; summary: { totalCapacity: number; totalDemand: number; isOvercommitted: boolean; maxOverBy: number; }; }; ``` Allocation rule (v1 simple): - distribute each milestone’s remainingPoints evenly across buckets between now and milestone endDate - if milestone has forecast end date, use that; else use milestone endDate - only include COMMITTED or IN_PROGRESS milestones (configurable) This is not perfect but explainable. --- ## Part C — API Add: - GET /api/capacity/plan?bundleIds=...&horizonWeeks=12 Returns: - bundlePlans[] - atRiskBundles[] sorted by maxOverBy - recommendedActions[]: - SCOPE_REDUCE for milestones driving overcommit - SLIP_MILESTONE suggestion if endDate too soon - ADD_CAPACITY suggestion if sustained overcommit Visibility: - obey bundle visibility rules; only return bundles user can view. --- ## Part D — UI (Program → Capacity tab) In Program Coordination: - add "Capacity" tab/panel Show: - At-risk bundles table: - total demand vs capacity - max overcommit - top driver milestones - Bundle capacity calendar table (not chart): - rows = bundles - columns = next N buckets - each cell shows demand/capacity and overBy indicator Clicking a cell opens modal: - drivers list with milestones and points Add quick links: - open milestone planning for a driver - open scope request creation (optional) --- ## Part E — Tests Add scripts/test-capacity-plan.ts: - seed bundles with capacities - seed milestones with remaining points and end dates - verify allocation math, overcommit detection, driver attribution - verify visibility filtering hides private bundle plans Add npm script: - test:capacity-plan Include in test:api. --- ## Part F — Docs Update docs/wiki/Modules-Dashboards.md or Program docs: - explain allocation rule and limitations - explain how to set bundle capacity --- ## Deliverables - bundle capacity config + admin UI - capacityPlanning service + /api/capacity/plan endpoint - Program Capacity tab with tables and drilldowns - tests + docs
