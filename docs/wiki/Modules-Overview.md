# Modules Overview

DeliveryHub is organized as a connected product, not as a loose collection of isolated tools.

Its major modules work together so that portfolio context, execution work, architecture governance, documentation, dashboards, and AI-assisted analysis all refer to the same delivery reality.

## The Main Product Areas

### Applications

The **Applications** module is the Application Portfolio Management layer.

It explains:

- which systems exist
- how they are grouped into bundles
- who owns them
- how they depend on each other
- what delivery impact they create

### Work Items

The **Work Items** module is the execution engine.

It manages:

- epics, features, stories, and tasks
- milestones and planning views
- risks and dependencies
- roadmap, forecast, and optimization workflows

### Architecture

The **Architecture** module is the design and review layer.

It manages:

- diagrams and architecture assets
- structured diagram creation
- architecture review cycles
- governance follow-through linked back to execution

### Wiki

The **Wiki** module is the knowledge and document management layer.

It manages:

- authored pages
- uploaded program artifacts
- reviewable documentation
- AI-assisted reading and summarization

### Dashboards

The **Dashboards** module is the executive visibility layer.

It turns operational data into portfolio, bundle, and milestone views that help leadership decide where to intervene.

### AI Insights

The **AI Insights** module is the assistive decision-support layer.

It provides:

- portfolio summaries
- evidence-backed Q&A
- trend analysis
- forecasting support
- risk propagation analysis
- what-if planning support

### Reviews

The **Reviews** capability spans modules rather than standing alone.

It is used in both Wiki and Architecture so governance activity becomes visible, traceable, and linked to follow-up work.

### Admin

The **Admin** module governs the shared rules of the system, including reference data, templates, ownership mapping, policy, AI controls, and operational tools.

## Why This Structure Matters

This modular structure gives DeliveryHub an advantage over disconnected point tools.

For example:

- a dashboard signal can drill into the exact work items behind a risk
- a document review can create follow-up work automatically
- an architecture decision can be tied to a bundle, milestone, and application
- an AI insight can point back to the work items, milestones, bundles, or reviews that support it

## Cross-Cutting Principles

Across modules, DeliveryHub is designed around a few consistent ideas:

- one connected source of operational truth
- visible ownership and accountability
- governance work that creates traceable follow-through
- AI that assists but does not silently make decisions
- documentation, planning, and execution staying linked together
