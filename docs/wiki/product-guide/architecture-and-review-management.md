# Architecture and Review Management

This capability is played by the **Architecture** module.

## What It Means

Architecture gives DeliveryHub its LeanIX-like and design-governance capabilities.

It is where teams store and review diagrams, integration views, and architecture artifacts that influence delivery.

## What Is Implemented

### Diagram gallery and structured creation

The module supports multiple diagram formats and a guided creation flow, including:

- blank diagram creation
- template-based creation
- uploaded architecture artifacts

The guided flow makes architecture work more consistent and easier to standardize across teams.

### Multiple diagram styles

Implemented support includes:

- Mermaid diagrams
- mind maps
- Draw.io and other uploaded diagram formats

This allows teams to work with architecture content in the format that is most useful for them.

### Metadata-driven architecture records

Diagrams can be linked to delivery context such as:

- bundle
- application
- milestone
- document type

This makes architecture part of the operating model, not a disconnected document library.

### Curated template support

Architecture teams can work from reusable templates so common diagrams start from an approved pattern rather than a blank canvas every time.

### Architecture reviews

Architecture artifacts can be submitted into a structured review cycle with:

- assigned reviewers
- review comments
- feedback flow
- linked execution follow-up

Review cycles are designed to support real back-and-forth, not just a one-time approval request.

### Review-linked work items

Each review request creates corresponding visible work so review effort is not hidden from the broader delivery picture.

### Reviewer guidance and structured feedback

The review experience supports more than free-form commentary. It is designed to preserve reviewer guidance, vendor response, and cycle history in a cleaner and more governable way.

### Comments and activity

Architecture changes and review discussions are captured through comments and events so there is a visible activity trail.

## Why These Features Matter

- architecture work becomes traceable instead of living in isolated files
- review cycles are connected to real execution work
- leadership can see that governance activity is happening, not just assume it is happening
- architecture decisions remain connected to bundles, applications, and delivery milestones

## Who Gets the Most Value

### Architects

Use this module to create, organize, and review design artifacts in a way that ties into delivery.

### Delivery leaders

Use architecture reviews to understand whether technical governance is delaying, enabling, or reshaping delivery.

### Executives

Use the architecture-linked review process as evidence that critical design and migration decisions are being governed, not handled informally.

## How Executives Should Use This in Practice

### CIO

Use architecture review visibility to confirm that critical migration designs are being challenged and followed through, not simply uploaded and forgotten.

### VP

Check which bundles have active architecture reviews, whether feedback is slowing delivery, and where decisions may need escalation.

### PMO

Use review-linked work items and activity history to monitor whether governance actions are becoming real execution tasks.

### Architects

Create diagrams from consistent templates, run formal review cycles, and keep decision history tied to the artifact instead of scattered across tools.

### Delivery managers

Use architecture review status to anticipate downstream delivery impact and make sure review feedback is visible in the work plan.
